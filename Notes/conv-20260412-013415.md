### 핵심 내용

[[Andrej Karpathy]]가 제안한 [[LLM Wiki]]의 핵심: LLM이 당신 대신 위키를 작성하고 유지보수하는 개인 지식 시스템.

**RAG vs LLM Wiki — 컴파일 vs 인터프리트:**

| RAG (기존) | LLM Wiki (제안) |
|---|---|
| 질문할 때마다 원본에서 검색 | 원본을 읽고 미리 정리해놓음 |
| 지식이 매번 휘발됨 | 지식이 축적되고 복리로 성장 |
| 교차 참조 없음 | 문서 간 링크와 관계가 이미 구축됨 |
| 모순점 발견 못함 | 새 정보가 기존과 충돌하면 표시 |

**3층 구조:**
1. **Raw Sources** — 원본 소스 (불변, LLM은 읽기만)
2. **Wiki** — LLM이 생성/관리하는 마크다운 파일들
3. **Schema** — CLAUDE.md 같은 운영 규칙 파일

**3가지 작업:** Ingest(소스 수집 → 위키 업데이트), Query(질의 → 좋은 답변은 위키에 저장), Lint(건강 체크 → 모순/고아 페이지 발견)

### 배경 지식
- Vannevar Bush의 **Memex(1945)** — "연관 경로가 있는 개인 지식 저장소"의 원조 비전. Bush가 풀지 못한 "누가 유지보수하느냐"를 LLM이 해결
- 비유: **"Obsidian이 IDE, LLM이 프로그래머, Wiki가 코드베이스"** — Karpathy 본인의 표현
- `index.md` + `log.md`로 ~100소스, ~수백 페이지 규모에서 **임베딩 RAG 인프라 없이** 작동

> [!insight] ★ Insight
> - "컴파일 vs 인터프리트" 프레임이 핵심 — 컴파일된 프로그램이 더 빠르듯, 컴파일된 지식이 더 깊은 답변을 줌
> - Obsidian이 이상적인 이유 — 마크다운 기반이라 LLM이 직접 읽고 쓸 수 있고, 그래프 뷰로 시각화, git 버전 관리 가능

## 2. 나의 맥락 — 왜 이것이 필요했는가

### 질문 (원문)
> 내가 지식을 탐구하는 패턴은 이렇게 llm과의 대화도 있을거고 웹사이트 가 일반적일거야
> 난 석사,박사 과정을 밟지 않아서 논문이라는 매체와 그리 친숙하지 않아
> 그리고 독서를 하다가 특정 영역이 맘에 드는 구절이 있을수 있고, 회사생활을 예로 들면 업무관련 지식을 위키 등에 적는 행동이 내가 생산하는 텍스트 인거 같아 뭐 데일리 업무 일지도 그에 속할 수 있겠지
> 하지만 어떤 학습을 하면 좀 나만의 위키에 축적이 되면 좋겠다 생각했었어 나만의 탐구 주제였지
> 그런에 이번에 이 제안을 보게 된거야
> 너무 흥미롭고 신나는 주제야 예를 들어서 para 라는 개념과 제텔카스텐 이라는 키워드도 이번 아티클로 시작해서 알게된 개념이기도 해
> 제텔카스텐 처럼 상향식으로 지식을 탐구하며 이렇게 llm-wiki로 나아간다면 너와의 모든 대화를 나만의 wiki에 정리할수 있게 된다면 너무 즐거울거 같아
> 이 맥락을 담아서 claude code 와 obsidiom cli 등 쓸수 있는 모든 도구 둘을 써서 그 환경을 구축해보고 싶어
> 예제로 코틀린 언어를 이런식으로 학습해 가는 과정을 진행해보고 싶어

### 핵심 내용

**나의 지식 소스 패턴:**
- LLM과의 대화
- 웹사이트 (주 소스)
- 독서 발췌
- 업무 위키, 데일리 로그

**핵심 고통:** 메모가 늘어나도 정리되지 않는 좌절감. 나만의 위키 정리 체계를 갖는 것이 크나큰 소망이었음.

**이번 아티클을 통해 새로 알게 된 개념:** [[PARA]], [[Zettelkasten]]

**첫 테스트 도메인:** [[Kotlin]] 언어 학습

### 핵심 내용: 세 가지 철학의 종합

**[[Steph Ango]]의 핵심 원칙:**
- 상향식(bottom-up) 접근
- 폴더 최소화 — "많은 노트가 하나 이상의 사고 영역에 속한다"
- `categories` frontmatter + [[Obsidian Bases]]로 **가상 분류**
- 날짜: `YYYY-MM-DD`, 태그/카테고리: 항상 복수형

**[[Zettelkasten]]의 노트 성숙도 흐름:**
```
Fleeting (Daily/, Notes/)  →  Literature (Clippings/)  →  Permanent (Wiki/)
    내가 쓴다                    내가 수집한다               LLM이 정제한다
```

**[[PARA]]는 폴더가 아닌 properties로:**
```yaml
status: active      # Project (진행 중)
status: ongoing     # Area (지속적)
status: archived    # Archive (완료)
```

### 최종 디렉토리 구조

```
workspace-obsidian/
├── CLAUDE.md            ← 🔧 LLM Wiki 스키마
├── Clippings/           ← 📦 원본 소스 (불변)
├── Notes/               ← 📝 내가 쓰는 노트
├── Wiki/                ← 📚 LLM이 관리하는 위키
│   ├── index.md         ←   전체 목차
│   └── log.md           ←   활동 로그
├── Daily/               ← 📅 데일리 노트
├── Categories/          ← 📂 Bases 뷰 (가상 분류)
├── Templates/           ← 📋 템플릿
│   └── Bases/           ←   .base 뷰 정의 파일
└── Attachments/         ← 📎 이미지/파일
```

**각 폴더의 출처:**

| 폴더 | 출처 | 역할 |
|------|------|------|
| `Clippings/` | Karpathy(raw) + Ango(Clippings) | 원본 소스 보관 (불변) |
| `Wiki/` | Karpathy(wiki) | LLM이 관리하는 컴파일된 지식 |
| `Notes/` | Ango(Notes) + Zettelkasten(fleeting) | 내가 직접 쓰는 노트 |
| `Daily/` | Ango(Daily) | 데일리 노트 |
| `Categories/` | Ango(Categories) | Bases 뷰로 가상 분류 |
| `Templates/` | Ango(Templates) | 노트 템플릿 |
| `CLAUDE.md` | Karpathy(schema) | LLM Wiki 운영 규칙 |

### 배경 지식

**왜 PARA를 폴더로 구현하면 안 되는가:** Project가 끝나서 Archive 폴더로 "이동"하면 모든 `[[wikilink]]`가 깨진다. properties로 `status: archived`만 바꾸면 파일은 그대로이고 Bases 뷰에서 자동 필터링된다.

**Categories는 가상이다:** `Categories/Books.md` 파일 내용은 `![[Books.base]]` 한 줄. 실제 분류는 각 노트의 `categories: ["[[Books]]"]` frontmatter가 하고, Bases가 이를 모아서 테이블로 보여준다.

> [!insight] ★ Insight
> 1. Ango의 가장 강력한 아이디어는 "Categories는 가상". 물리 폴더를 바꾸지 않고도 분류 체계를 자유롭게 진화 가능
> 2. LLM Wiki에서 가장 중요한 경계선은 "누가 쓰는가" — Clippings+Notes(사람) vs Wiki(LLM)
> 3. PARA를 폴더로 구현하면 실패하기 쉬움 — properties 기반이 올바른 접근

## 5. CLAUDE.md — LLM Wiki의 두뇌

### 질문 (원문)
> 추가로 궁금한건 CLAUDE.md 는 .claude 폴더 하위에 있어야 하는거 아니야?

### 핵심 내용

**CLAUDE.md는 2가지 레벨로 작동한다:**

| 위치 | 범위 | 역할 |
|------|------|------|
| `~/.claude/CLAUDE.md` | 글로벌 — 모든 프로젝트 | 언어, 이모지 등 범용 규칙 |
| `프로젝트루트/CLAUDE.md` | 프로젝트 — 해당 디렉토리만 | LLM Wiki 스키마 |

세션 시작 시 **두 파일을 모두 자동 로드**. 글로벌 규칙 위에 프로젝트 규칙이 합쳐진다. vault 루트의 `CLAUDE.md`에 위키 운영 규칙을 넣는 것이 Karpathy가 말한 "스키마" 역할을 정확히 수행.

**CLAUDE.md에 정의한 주요 규칙:**
- Vault 구조 명세 (각 디렉토리 역할)
- 마크다운 규칙 (wikilink 필수, YAML frontmatter 필수, 날짜 YYYY-MM-DD)
- Frontmatter 규칙 (categories, type, domain, related, sources, created, updated)
- Ingest/Query/Lint 워크플로우 상세 절차
- 위키 페이지 명명 규칙
- 핵심 원칙: Clippings 불변, 모순은 callout으로 표시, 최소 1개 wikilink 연결
