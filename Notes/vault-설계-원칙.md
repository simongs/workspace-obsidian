# Vault 설계 원칙

> LLM-wiki를 길게 운영하기 위한 스스로의 원칙. 계속 수정·추가될 가변 노트.

## 1. 쓰는 고통이 느껴진 뒤에 넣는다

**좋아 보여서 넣는 기능은 대개 쓰이지 않는다.** 현재 쓰면서 구체적 불편을 느낀 뒤에 도입해야 유지된다.

- Bases? Reference 지식 관리엔 지금 불필요. 아티클·소감이 20개 쌓이고 검색이 잦아질 때 도입.
- 새 폴더? 기존 폴더 안의 섹션으로 먼저 시도. 분리 필요가 명확해지면 그때.
- 새 모드/스킬? 기존 모드 확장으로 커버되는지 먼저.

## 2. 새 구조보다 기존 구조 확장

새 폴더·새 타입·새 모드는 **유지 부담**이다. 기존 자리에 섹션 추가가 먼저.

## 3. 운영하면서 체계가 자라게 한다

처음부터 완벽한 체계를 갖추려 하지 않는다. kepano vault도 처음부터 31개 Bases가 아니었다. **필요마다 하나씩** 쌓였다.

## 4. AI 제안에 브레이크를 걸 권리는 항상 내게 있다

Claude가 좋은 옵션을 계속 쌓을 때, "이거 과잉 아닌가" 판단이 들면 **그게 건강 신호**다. Evan Moon 글의 "AI에 끌려다니는 것 vs AI를 잘 쓰는 것"의 차이는 바로 이 판단력.

## 5. 두 운영 영역 (암묵적)

- **Karpathy 영역** — `Wiki/`, `Sources/docs/`. LLM이 top-down으로 운영
- **사람 영역** — `Notes/`, `Notes/Inbox/`, `Sources/articles/` (예정). 내가 bottom-up으로 쓰고 고침

명시 체계화는 하지 않는다. 필요할 때 이 두 영역이 자연스럽게 분리되도록만 둔다.

## 6. 재검토는 합리적이다

3-6개월 동안 골격을 조정하다 본격 운영하는 쪽이, 6개월 뒤에 "이걸 왜 이렇게 했지" 하는 것보다 낫다. 초반 재검토는 건강한 습관.
