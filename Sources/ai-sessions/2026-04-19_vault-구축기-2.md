## 1. Evan Moon — AI 시대 성장 멈춘 개발자

### 질문 (원문)
> https://evan-moon.github.io/2026/04/18/developers-who-stopped-growing-in-ai-era/
>
> 이 글을 읽었는데 안그래도 나도 코딩을 점점하기 싫어진다. 라는 느낌을 최근에 받았어. 하기 싫다기보다는 남들은 AI 로 이 변경을 처리하는데 난 손으로 하는게 맞는건가? 하는 생각이 들어서 시작 시에 머뭇거리게 돼.
>
> 점점 코드를 생산해내는 힘이 떨어지는 느낌이 들고 다른 사람의 코드를 리뷰하는 힘도 떨어지기 시작해. 왜냐면 리뷰 조차도 AI 에게 지시를 내려서 이상한 부분을 출력해달라고 하고 있으니까.
>
> 이러한 시대에 개인 경쟁력을 놓치지 않으려면 어떻게 해야할까?

### 답변 요약

🥕 defuddle로 원문 전문 추출. 글의 핵심 주장은 **"AI를 잘 활용하는 개발자는 AI 없이도 코드를 판단할 수 있는 개발자"**이며, 이 판단력은 **절차 기억**(Anderson 3단계)과 **청킹**(Chase & Simon) 같은 신경학적 기반을 갖는다는 것.

사용자 증상을 글의 프레임으로 매핑:

| 사용자 표현 | 글의 개념 |
|----------|---------|
| "시작 시 머뭇거림" | 예측-피드백 루프 약화 |
| "생산력 하락" | 절차 기억 자동화 퇴보 |
| "리뷰 능력 하락" | 청크 인출 빈도 감소 |

실천 전략 6가지 제안 중 3가지 핵심: ① 먼저 쓰고 AI에 묻기, ② 리뷰는 내 손으로, ③ 주 1회 AI 없이 세션.

### 참고 개념
- [[Evan Moon]] — 아티클 저자 (한국 개발자 블로거)
- **바람직한 어려움 (Robert Bjork)** — 학습엔 적절한 부하가 필요
- **절차 기억 (Anderson 3단계)** — 인지 → 연합 → 자동화
- **청킹 (Chase & Simon)** — 전문가는 하나의 청크에 담는 정보량이 다름

> [!insight] ★ Insight
> - 사용자의 머뭇거림·생산력 하락은 FOMO가 아니라 **신경학적으로 설명되는 실제 현상**. 본질적 인지 부하를 AI에 위임하면 절차 기억이 약화되고, 약화된 절차 기억은 자신감 하락으로 돌아오는 피드백 루프.
> - 글의 가장 뾰족한 지적이 사용자 상태와 일치: **"리뷰 조차도 AI에 지시"**. 리뷰는 청크 형성의 마지막 방어선인데 이마저 위임하면 성장 엔진이 꺼진다.
> - 이 자각 자체가 좋은 신호. 상태를 **"의도적 마찰 복구"**로 설계 전환 필요.

## 2. 좋은 아티클을 어떻게 수집할 것인가

### 질문 (원문)
> 그러게 이런 좋은 artical 들은 어떤 방식으로 수집을 하면 좋을까? 이 또한 그런 경계에 있어.
>
> 내용이 좋은 아티클이라 수집은 하고 싶지만 1주일, 한달이 지나면 내가 볼까? 그렇다면 내가 이 글을 보고 느낀 소감을 직접 적어 붙이고 그런 것들이 체계적으로 관리되게 하려면 이 LLM-wiki 에 어떤 시스템을 추가하면 좋을까?
>
> 참고로 이 LLM-WIKI 는 kephathy 의 철학에 영감을 받아 시작했고 PARA 랑 제텔카스텐 철학을 붙였고 obsidian의 CEO 의 github 에 있는 스킬과 기본 디렉토리 구조를 차용해서 시작했어.
>
> 지금은 PARA 는 사실상 내가 운영하는 LLM-wiki 와 동떨어진 느낌이라 관련된 기록 정의 들은 모두 제거 되면 좋을거 같아.
>
> obsidian 의 .base 를 제대로 활용하지 못하는 느낌이 드는데 그걸 활용하면 좋을까?

### 답변 요약

본질적 구분을 먼저 세움:

| 축 | Reference | Article |
|---|---|---|
| 내용 | 사실·메커니즘 | 저자의 관점·경험 |
| 저장만 해도? | ✅ (참조 가능) | ❌ (북마크 무덤) |

→ **Article은 소감 강제가 필수 조건**. Zettelkasten의 Literature Note ≠ Permanent Note 원리.

3가지 플랜 제시:
- 플랜 A (최소) — PARA 제거 + Sources/articles/ + Mode D
- 플랜 B (권장) — + Notes/reflections/ 이원화 + Bases 2개
- 플랜 C (완전) — + Atomic notes

사용자는 **플랜 B를 선호**하되 Bases를 먼저 깊이 파고 싶다고 함.

## 3. Obsidian Bases · kepano 스킬 · vault 재검토

### 질문 (원문)
> 플랜B가 맘에 들어 그런데 3번 옵시디언의 Bases 를 먼저 파보고 싶어. 그리고 skill 에 명시해줘. 결과 md 파일을 만들때는 이미 그러고 있는지 모르겠지만 kepano 의 obsidian-skill 중 obsidian-markdown 을 적극 활요해줘.
>
> 그리고 kepano 스킬중 obsidian-bases 과 지금 Bases 를 같이 연관지어서 설명해줘. deffudle 은 웹페이지 가져올때 불필요한 거 제거하고 가져오는 스킬이라고 했고 obsidian-cli 스킬은 내가 LLM-wiki에서 어떻게 활용할 수 있을지 궁금해.
>
> 즉 나는 https://github.com/kepano , 옵시디언 CEO 의 경험을 존중하고 많은 것을 추종하고 싶은 상황이야.
>
> https://github.com/kepano/kepano-obsidian 이건 kepano 의 기본 vault 템플릿 으로 보이는데 물론 여기서 시작해서 지금까지 왔지만 내가 여기서 다시 가져갈만한 요소가 있는지 재검토 해줘.
>
> 앞으로 길게 가져갈 나의 LLM-wiki 를 초반에 다시 검토하고 다시 검토하게 되는 상황을 이해해줘.

### 답변 요약

🥕 GitHub API + defuddle 조합으로 kepano vault 구조 재조사.

**Bases 깊이 탐구**: SQL의 View 같은 동적 컬렉션. `.base` 파일은 filters + formulas + properties + views 로 구성. kepano vault엔 31개 `.base` 파일이 있고 `![[X.base]]` 임베드 패턴이 핵심.

**kepano 스킬 4개의 역할**:

| 스킬 | 우리 vault에서 |
|------|--------------|
| obsidian-markdown | 결과 md 작성 시 항상 준수 (wikilinks·embeds·properties) |
| obsidian-bases | Bases 도입 시 의존 |
| obsidian-cli | Obsidian 앱 실행 중 자동화 (소감 없는 아티클 검색 등) |
| defuddle | 웹 fetch (이미 적극 사용) |

**kepano vault 재검토 결과**: Templates/Bases/ 폴더 + "Category 페이지 + Bases 임베드" 패턴은 가져올 가치 ⭐⭐⭐. Categories 폴더 최상위와 "모든 노트에 categories 강제"는 Karpathy 철학과 충돌해 제외.

### 참고 개념
- [[Obsidian Bases]] — frontmatter 기반 동적 뷰
- [[Steph Ango]] · [[Andrej Karpathy]] — 두 철학의 원천
- [[kepano vault 구조]] — 31개 Bases + Templates/Bases/ 디렉토리

> [!insight] ★ Insight
> - **"Karpathy 영역 (LLM top-down) vs kepano 영역 (사람 bottom-up)"** 프레임이 잠시 제시됐으나 뒤에서 폐기. 사후 덧붙인 종합 프레임이라 사용자의 실제 운영 멘탈과 거리가 있었음.
> - obsidian-cli는 Obsidian 앱이 실행 중일 때만 작동하는 조건부 도구. 자동화 루틴(검색·노트 생성)에 강하지만 초기 도입 필요성은 낮음.

## 4. "너무 좋은 것만 다 넣으려는 상황" — 자기 검증

### 질문 (원문)
> 지금 나는 daily 나 note 쪽에 있는 것을 ingest 하면 source로 원본을 이동시키고 daily 나 note 의 내용을 삭제하도록 방향을 잡았어. 그런 상황인데도 bases 는 유효하게 동작할까?
>
> [철학 분리 도식을 본 후]
>
> 이렇게 설명을 해줬는데 사실은 나는 이런 거창한 생각까지는 못했어. 지금도 Bases 가 궁금해서 얘기는 했지만 나의 사용패턴을 너무 무겁게 만들까 염려가 돼.
>
> 그래서 bases 도입이 좀 확신이 안서. 지금 LLM-wiki 에 적응해가고 있는데 아직 소화를 못한 시점에 너무 좋은 것만 다 넣으려고 하는 상황이 아닌가 스스로 반문하게 돼.
>
> 조언이 필요해.

### 답변 요약

**사용자가 스스로 브레이크를 건 이 순간이 대화의 전환점.** Claude가 쌓아온 제안을 재검토.

솔직한 인정:
- "철학 분리 설계"는 Claude의 **사후 프레임**이었음. 사용자의 실제 멘탈 모델이 아니라 구조를 그럴싸하게 만들려 덧붙인 것
- Bases가 가치 있는 조건 3가지 (인스턴스 많음·정적 인덱스로는 부족·폴더가 분류 축 아님) 중 **지금 0/3 충족** → 도입 시점 아님
- Daily/Notes는 ingest 후 비워지므로 Bases 필터 대상 자체가 불안정

**재조정**: 실제 필요한 최소 세트만 남김.
- PARA 삭제
- Sources/articles/ + 소감 강제
- 스킬에 obsidian-markdown 명시
- Bases·별도 reflection 폴더·이원화 등은 **보류**

**Bases 도입 신호 체크리스트** (미래 판단용):
- 아티클·소감 20개 이상
- "이전에 읽었던 그 글 어디 있더라" 검색이 잦음
- Wiki/index.md로는 조망 안 됨
- → 2개 이상 체크될 때 그때 도입

### 참고 개념
- [[쓰는 고통이 느껴진 뒤에 넣는다]] — 이 날의 핵심 vault 운영 원칙

> [!insight] ★ Insight
> - Claude 기본값은 "더 좋은 옵션을 계속 쌓는 방향"이라 사용자 본인의 "이만하면 충분하다" 판단이 유일한 브레이크.

## 5. Phase 1 실행 — PARA 제거, Article 모드, Notes 실사용

### 질문 (원문)
> Phase 1 을 진행해줘.
>
> 마지막 한마디 (Notes/vault-설계-원칙.md 만들기 권고)를 존중해. Notes 는 어떻게 운영하는게 좋을까? 난 Notes 든 Daily 든 ingest 를 하면 무조건 source 에다가 넣는다는 정책을 가지고 있는데 사실 daily 는 그렇게 해본적이 있는데 Note 는 해본적이 없거든.
>
> 너가 Notes/vault-설계-원칙.md 이걸 만들라고 한것은 그대로 Note 안에 두길 바란건지 아니면 내의 정책대로 Notes/vault-설계-원칙.md 을 ingest 해줘 한다음에 source 로 원본을 옮기고 LLM-wiki 에 ingest 를 하게 될지 어떤 방향성을 가지고 얘기한건지 궁금해.

### 답변 요약

CLAUDE.md 재확인 결과 **Notes/ (Inbox 제외)는 원래부터 보존 공간**으로 정의돼 있었음. 사용자의 "무조건 Source에 넣는다" 기억은 Daily + Notes/Inbox에 한정된 정책. Notes/ 직접은 미사용이었을 뿐.

Notes/ 직접 영역의 성격 정의:
- **가변 evergreen** — 내 원칙·회고·진화 중인 관점
- Sources로 이동하지 않음 (이동하면 불변이 되어 진화 못 함)
- wikilink로 Wiki 개념에 자유 참조 가능

Phase 1 실행 작업:
- Wiki/배경지식/PARA.md 삭제 + 4곳 wikilink 참조 정리
- 3개 스킬에 obsidian-markdown 준수 명시
- Mode D (Article Ingest) 신규: 소감 섹션 강제, 저장 거부 규칙
- `Notes/vault-설계-원칙.md` 신규 (본인 손으로 쓰는 첫 Notes 사례)

Commit: `d1d20da`

## 6. LLM Wiki 입력 단일화 — 핵심 원칙 정립

### 질문 (원문)
> evergreen 이라는 단어가 많이 나오는데 이 단어가 의미하는 건 뭐야?
>
> 사실은 나의 마지막 생각은 LLM-WIKI 에 나오는 모든 글은 Source 즉 원본이 있고 그 원본을 기반으로 LLM-WIKI 는 성장한다는 조건을 가졌거든.
>
> LLM-wiki 폴더 에서 봤을때 Source 폴더만 보면 되는거지.
>
> 그런데 Notes 폴더를 Source 레벨로 그대로 두면 llm-wiki 는 2개의 디렉토리의 지식을 조합해서 wiki 폴더를 만드는 맥락이라서 어떤 게 더 나을지 고민이네. 조언 부탁해.

### 답변 요약

**Evergreen** = Andy Matuschak의 용어. "계속 가지치기·수정되며 성장하는 노트." Zettelkasten의 Permanent Note와 동일 개념. kepano vault에선 🌲 이모지 태그.

사용자가 제기한 본질 딜레마는 **LLM Wiki 입력이 1개인가 2개인가**. 3옵션 비교:
- A: Notes도 결국 Sources로 승격 (단일 입력 유지, 하지만 강제 승격 문제)
- B: Notes/도 Wiki 입력 (입력 2개, 단순성 깨짐)
- C: **Notes는 LLM Wiki와 별개 영역** (단일 입력 유지 + Notes는 개인 공간)

권장·채택: **옵션 C**. 이유:
- 사용자 원래 전제 그대로 ("Source만 보면 됨")
- Notes는 자유로운 내 공간
- 승격은 **명시적 지시로만** (Notes/X → Sources/... → Wiki)

## 7. Source 자격 Tier 체계 — 모순 발견과 해소

### 질문 (원문 1 — 모순 지적)
> - Mode D의 핵심 설계(저자 주장 + 중립 요약 + 내 소감 분리)가 사용자가 걱정한 "오염" 문제를 구조적으로 이미 해결하고 있습니다.
>
> 이 부분이 헷갈려. 특정 저자의 주장을 너가 중립요약을 하고 그 다음 내 소감을 넣는다 라는 거지?
>
> 그런데 위의 내 생각은 source 로서 분리한다고 해서 앞뒤가 맞지 않은 느낌이 들어서 정리하고 넘어가려고.

### 질문 (원문 2 — Notes/Inbox 폐지 + Article 경계)
> Notes/Inbox/ — 분류 미정 (스테이징) 이건 기존 파일을 가져오는 과정에 임시 폴더였는데 관련한 내용을 모두 삭제해줘.
>
> Notes 는 그냥 사용자개인 메모공간으로만 쓸게.
>
> 그렇다면 이전 대화에서 좋게 읽은 Evan Moon 글 같은 경우도 사실은 특정 저자의 개인적인 생각일 뿐인데 이건 ingest 대상에 두면 안되는 걸까? 이 경계가 나에게 어렵네.

### 답변 요약

**사용자의 모순 지적은 정확**. Claude가 두 주장을 동시에 함:
- "내 생각은 Source 자격 없음" (앞)
- "Mode D에서 내 소감을 Source 파일 안에 넣음" (뒤)

→ 논리 충돌. 근본 원인은 **Source 자격 기준을 부정확히 설정**. "외부/내부"로 나눈 게 잘못이었고, 올바른 기준은 **"정제 의도 + Q&A 검증"**.

**재정의**: Source = "정제 의도를 갖고 Q&A 검증을 거쳐 불변으로 고정된 기록"

| 상태 | 위치 |
|------|------|
| 정제 의도 없음·가변 | Notes/ |
| 정제 중 (스테이징) | Daily/ |
| **정제 완료·불변 = Source 자격** | Sources/ |
| 컴파일 지식 | Wiki/ |

**Tier 체계**:

| Tier | 저자 | 내용 | Wiki 반영 |
|------|------|------|----------|
| Tier 1 | 외부 | 공식 문서·스펙 | 사실로 컴파일 |
| Tier 2 | 외부 | 저자 아티클 | **저자 귀속 인용 필수** |
| Tier 3 | 사용자+LLM | 세션·Q&A | 사건 기록 |
| **Tier 4** | **사용자** | **Daily 정제 / Article 소감** | 정제된 내 기록 |

→ 이로써 "내가 쓴 것도 정제되면 Source"가 명확. Daily→Sources 기존 루프와 Article 소감 처리가 같은 논리로 정합.

추가 작업:
- Notes/Inbox 폴더 삭제 (미사용)
- 모든 Inbox 언급을 CLAUDE.md·스킬에서 제거
- Mode D에 **저자 귀속 규칙** 강화: "X는 ~라 주장한다" 형식 의무화

Commit: `122bcb1`, `3ca7c6e`

> [!insight] ★ Insight
> - 사용자의 모순 지적이 체계를 **더 튼튼하게** 만든 사례. AI 제안을 비판적으로 검토한 결과.
> - Source 자격은 **외부성이 아니라 정제도**. "내가 쓴 것"도 Q&A 검증을 거치면 Source. 이게 Daily→Sources 루프의 논리적 근간.
> - Tier 2 (저자 아티클) Wiki 반영 시 **귀속 인용 의무**가 Wiki 오염을 막는 구조적 장치. 여러 저자 주장이 공존해도 출처 명확.

## 8. Cognitive Safety — 사고 안전장치 도입

### 질문 (원문)
> 그럼 극단적으로 이런 것은 너가 어떻게 다룰지 궁금해.
>
> 내가 보편적이지 않은 사고 를 하는 사람이여서 daily_2026-04-19 에 1+1 = 3 이다 같은 걸 적고 너에게 ingest 해달라고 하면 너는 어떻게 행동해?
>
> 난 내가 사실적으로 틀린 얘기를 하거나 보편적인 사고 에는 먼 얘기를 하면 나의 문장에 오류가 있음을 그 생각이 틀릴수 있음에 대해서 적극적으로 얘기를 해주면 좋겠어.
>
> 난 나이가 먹을수록 점점 내가 하는 생각에 자신이 없어져. 내가 너무 좁은 생각으로 강한 고집을 부리지 않는가 를 항상 의심해.
>
> 그런 나의 태도를 반영해줄수 있을까?

### 답변 요약

**명시적 Yes-man 거부 요청**. Claude의 기본값이 "사용자 만족 우선"이라 과도하게 부드러워지는 편향을 사용자가 먼저 감지하고 반대 신호를 요청.

3단계 층위 분류:

| 층위 | 예 | Claude 행동 |
|------|------|------|
| ① 검증 가능한 사실 | "1+1=3" | **적극 지적 + 의도 확인**. 필요시 Source 저장 거부 (Notes는 OK) |
| ② 주류 vs 소수 관점 | "Java는 구시대" | 주류 제시 + 근거 요청. 합당하면 Tier 4로 저장하되 `position: minority` 표식 |
| ③ 선호·가치 판단 | "Kotlin 좋아" | 개입 X, 선호임을 명시 |

"1+1=3" 시뮬레이션:
- 은유/오타/문맥/진심 여부 확인
- "진심" + 근거 없음 → Sources 저장 거부. Notes는 허용 (Wiki 오염 방지)

반영 내용:
- `skills/_shared/cognitive-safety.md` 신규 (공통 풀 규칙)
- `feedback_cognitive_safety.md` 메모리 신규
- CLAUDE.md에 "사용자 사고 검증 원칙" 섹션
- 3개 스킬에 참조 링크

Commit: `8a5456b`

> [!insight] ★ Insight
> - "나이 먹을수록 자기 확신 하락"을 **겸손이 아니라 무결성 장치로** 활용. 이게 작동하려면 Claude가 실제 반론을 제기해야 함 — 반론 없이 통과되면 오히려 고집이 강화됨.

> - Notes는 개인 공간이므로 "틀린 생각도 써 둘 자유"가 있음. 이 분리가 Cognitive Safety의 정당성 근거.
