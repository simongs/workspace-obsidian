# obsidian-learn-ingest

소스를 읽고 바로 wiki를 쓰는 대신, 사용자와 대화로 이해를 맞춘 다음 wiki를 작성한다.

**이 스킬이 존재하는 이유**: LLM이 wiki를 만드는 동안 사람은 구경꾼이 된다. 기존 wiki 지식을 소스에 섞어버리는 실수도 생긴다. 대화를 통한 교차검증이 wiki 품질과 사용자의 실제 학습을 동시에 높인다.

**핵심 철학**: 소스가 부실하면 그대로 wiki로 옮기지 않는다. 먼저 사용자의 이해를 높이고 원본을 보강한 뒤, wiki로 가도 충분한 수준이 됐을 때 ingest를 진행한다.

## 소스 유형 판별 (스킬 진입 시 가장 먼저)

소스가 무엇인지 파악하고 적절한 모드를 선택한다.

| 소스 유형 | 판별 기준 | 사용 모드 |
|-----------|-----------|-----------|
| 이미지, 다이어그램, 일반 문서, 노트 | 파일 경로 / 텍스트 | **Mode A: Document Ingest** (기본) |
| GitHub 레포 / 기술 소개 URL | URL + 기술 학습 목적 | **Mode B: Tech Progressive** |
| 긴 공식 문서 / 레퍼런스 URL | URL + "묶음씩 번역", "같이 읽어가며", "Insight 붙여서" 등 | **Mode C: Chunked Translation Review** |

> URL이 주어졌을 때 "그냥 정리만" → Mode A, "새로 배우고 싶어" → Mode B, "묶음씩 번역하며 리뷰" → Mode C.
> 소스가 이미 Sources에 있다면 → Mode A로 바로 진입한다.
> **기본은 Mode A**. 사용자가 명시적으로 Mode B/C를 요청하지 않으면 Mode A로 진행.

### 기존 Sources·Wiki 자동 인지 (모든 모드 공통)

**스킬 진입 즉시**, 사용자가 말하기 전에 다음을 수행한다:

1. 사용자 요청의 **핵심 키워드/도메인** 추출 (예: "Zuora 구독", "Refund", "Kotlin 코루틴")
2. `Sources/` · `Wiki/` 에서 해당 키워드로 **기존 파일이 있는지 빠른 grep/glob**
3. 있으면 사용자에게 알림:

```
이 주제와 관련된 기존 자료를 발견했습니다:
- [[Sources/docs/zuora/...]]  (이미 ingest됨)
- [[Wiki/북미/Zuora/...]]     (기존 페이지)

진행 방식을 선택해주세요:
(a) 기존 Wiki 보강 — 새 Source 추가 + 기존 Wiki 업데이트
(b) 종합 페이지 신규 — 기존 Sources 여러 개를 참조해 비교·종합 Wiki 생성
(c) 무관, 새 주제로 ingest — 동일 키워드지만 별개 내용
```

**왜 필요한가**: 사용자는 vault 상태를 모두 기억하고 질문하지 않는다. Claude가 먼저 확인해 중복 ingest를 막고, 종합 페이지(비교·플로우) 기회를 포착한다.

### 페이스 변수 (모든 모드 공통)

사용자가 시작 시 다음 중 하나를 지정하면 모든 모드에 적용된다. 무지정 시 기본값 `step`.

| 변수 | 동작 |
|------|------|
| **`step`** (default) | 단계마다 사용자 확인 대기 (현재 기본 동작) |
| **`rush`** | 모든 단계를 한 번에 제시, 중간 대기 없음. 마지막에만 리뷰 |
| **`partial`** | 중간까지만 진행 (Mode B: Layer 2까지 / Mode C: TOC 절반 / Mode A: 분석 요약만) |

사용자 표현 매핑:
- "한 번에 쭉 보여줘", "전부 쭉" → `rush`
- "반만 해보고 결정하자", "일단 절반만" → `partial`
- 명시 없음 → `step`

### 모드 판별이 애매할 때

소스 유형이나 사용자 의도가 명확하지 않으면 **직접 선택지를 제시하고 물어본다.** 임의로 모드를 선택하지 않는다.

```
어떤 방식으로 진행할까요?

**Mode A — 문서 정리** (기본, 가장 빠름)
Claude가 소스를 분석 → 1차 정리 제시 → 사용자가 리뷰 → 수정 반영 후 Sources + Wiki 작성.
이미 아는 내용을 기록하거나, 단순 정리가 목적일 때.
페이스 변수(`step`/`rush`/`partial`)로 속도 조절 가능.

**Mode B — 기술 점진적 학습** (Bird's Eye View → 단계별 심화)
처음 접하는 기술이나 개념을 층위별로 이해하고 싶을 때.
Claude가 Layer 0(큰 그림)부터 시작해 원하는 깊이까지 안내.

**Mode C — 자동 묶음 번역 리뷰** (원문 보존 + Insight 포함)
긴 공식 문서를 Claude가 섹션·문단 구조에 따라 자동 묶음으로 잘라 번역·요약·Insight를 병치 제공 → 사용자가 리뷰.
덩어리째 훑기 + Claude 메모(default ON)로 본인 작업과의 연결 포인트까지 짚어줌.
```

### 링크 추적 규칙 (기본 동작)

소스 파일 안에 `[[다른 파일]]` 형태의 wikilink가 있으면 **반드시 해당 파일을 모두 따라가서 함께 ingest한다.**
사용자가 별도로 요청하지 않아도 기본 동작이다.

```
사용자가 A.md를 ingest 요청
  → A.md 안에 [[B.md]], [[C.md]] wikilink 발견
  → B.md, C.md 도 함께 읽어서 통합 분석
  → 하나의 Sources + Wiki로 병합 ingest
  → A, B, C 모두 삭제
```

예외: 링크가 이미 Sources나 Wiki로 연결된 경우 (이미 ingest된 문서) → 스킵

### 원본 소스 처리 규칙 (ingest 완료 후)

| 소스 위치 | 처리 방법 |
|-----------|---------|
| `Daily/` | **ingest 완료 후 파일 삭제** (Daily = 스테이징 영역. Source에 원본 전문 callout 포함) |
| `Notes/Inbox/` | **ingest 완료 후 파일 삭제** (처리 대기 스테이징 영역) |
| `Notes/` (Inbox 제외) | 파일 보존. 하단에 역참조 추가: `related: [[Sources/...]], [[Wiki/개념명]]` |

## 소스 품질 평가 (모든 모드 공통)

모드 진입 전, 소스를 읽은 뒤 wiki로 바로 옮겨도 충분한 수준인지 평가한다.

**부실한 소스의 징표**:
- 단편적인 메모, 키워드 나열, 미완성 문장
- "이게 뭔지 궁금하다", "나중에 정리" 같은 의도만 있고 내용이 없음
- 핵심 개념의 설명 없이 용어만 등장
- 작성자 본인도 완전히 이해하지 못한 상태로 남긴 흔적

**부실하다고 판단되면 → 보강 모드 진입**:

```
소스를 읽었습니다. 내용이 아직 단편적이어서 wiki로 바로 옮기기보다
먼저 이 주제를 함께 탐구한 뒤 더 풍부한 상태로 기록하는 게 좋겠습니다.

[주제에 대한 심화 설명 — Claude가 알고 있는 것을 풍부하게 가르쳐 줌]

이해가 됐으면 추가로 궁금한 점을 물어보세요.
충분히 이해됐다고 느껴지면 그때 Sources + Wiki를 작성하겠습니다.
```

보강 과정:
1. Claude가 먼저 주제를 폭넓게 설명 (소스에 없는 내용도 포함, 단 출처 명시)
2. 사용자가 이해를 확인하고 추가 질문
3. **"이 정도면 wiki로 가도 좋겠다"는 합의가 이뤄질 때** Phase 2 또는 해당 모드로 진행

**충분하다고 판단되면 → 해당 모드 본 단계로 바로 진행**.

## MODE A: Document Ingest Mode (기본)

**언제 사용**: 이미지, 다이어그램, Daily 노트, Notes, 일반 문서 등을 ingest할 때. 가장 자주 쓰이는 기본 모드.

### Phase 1: 소스 분석 (사용자에게 알리기 전에 조용히)

소스 파일을 Read 도구로 읽는다. 이미지라면 반드시 Read로 직접 열어서 시각적으로 확인한다.

읽으면서 스스로 정리:
- 이 소스가 다루는 **핵심 주제** 는 무엇인가?
- **섹션이나 구분**이 있다면 각각 무엇을 설명하는가?
- **모호하거나 확인이 필요한 부분**은 어디인가?
- 기존 vault wiki와 연결될 개념은 무엇인가?

> 주의: 이 단계에서는 기존 wiki 지식을 소스 내용에 섞지 않는다. 소스에 있는 내용만 정리한다.

→ 소스 품질 평가 후 충분하면 Phase 2로, 부실하면 보강 모드 진입.

### Phase 2: 1차 정리 제시 + 리뷰 & 수정 반영

분석한 내용을 사용자에게 제시한다. 형식은 소스 성격에 따라 다름:

**A. 섹션/번호가 있는 소스** (이미지·다이어그램·번호형 문서)
```
[소스 제목 / 파일명] 을 읽었습니다. 제가 이해한 내용을 섹션별로 설명드리겠습니다.

**① [섹션명]**
[이 섹션에서 이해한 내용. 이미지라면 다이어그램/화면에서 읽은 것만.]

**② [섹션명]**
[...]
```

**B. 섹션 없는 긴 텍스트 / 개념 문서** (권장 템플릿)
```
## [주제명] — 1차 정리

### 한 줄 정의
[소스 문구 기반으로 직관 요약]

### [주요 카테고리 표 or 구조]
| ... |

### 본인 프로젝트와의 관련성 (메타 주석)
[여기만 기존 wiki/작업 맥락 허용 — "소스 본문" 설명 아님]

### 다음 행동 선택지
1. ...
2. ...
```

**리뷰 요청 공통 문구** (두 형식 모두 끝에):
```
---
확인해주실 사항:
- 제가 잘못 이해한 부분이 있나요?
- 추가로 설명해주실 내용이 있나요?
- 이대로 Wiki로 저장해도 될까요?
```

**필요 시 사용자 응답에 따라 Q&A 이어감** — 사용자가 질문하지 않으면 바로 Wiki 저장으로 진행. "Q&A 진행"을 강제하지 않는다.

#### ⚠️ 소스 본문 설명 vs 메타 주석의 경계

이 규칙은 **Mode A의 핵심 안전장치**다 (스킬 최상단 "이 스킬이 존재하는 이유" 참조).

| 영역 | 외부 지식 허용? |
|------|---------------|
| **소스 본문 설명** (섹션 ①②, "한 줄 정의", "구조 표") | ❌ **금지**. 소스에 있는 것만 씀 |
| **메타 주석** (본인 작업 관련성, 다음 단계 선택지, 비교 각주) | ✅ 허용. 단 "(소스에 없음 — 기존 wiki/일반 지식)" 표식 권장 |

LLM이 소스 본문에 자기 배경 지식을 섞으면 ingest 품질이 망가진다. 경계를 섹션으로 분리한다.

### 팁: 이미지 소스일 때

이미지는 텍스트 문서보다 오독 가능성이 높다. Phase 2 A형식에서 반드시:
- 섹션 제목(■, ①, 번호 등)을 그대로 인용해서 설명
- 다이어그램의 흐름은 화살표 방향대로 순서를 설명
- 읽기 어려운 텍스트는 "이 부분은 명확히 읽히지 않았습니다" 라고 솔직하게 표시

## MODE B: Tech Progressive Learning Mode

**언제 사용**: 새로운 기술, 라이브러리, 도구, 프레임워크를 처음 접하는 경우.
기반 지식이 없어도 괜찮다. Bird's Eye View부터 시작해 단계적으로 이해한다.

> 이미 어느 정도 아는 기술이라도 "새로 공부하고 싶어"라면 Mode B 사용 가능.
> "그냥 README를 정리만 해줘"라면 Mode A.

### 사전 준비: 소스 확보

```
URL이 주어진 경우:
  1. WebFetch 또는 defuddle로 페이지를 읽는다
  2. Sources/github/YYYY-MM-DD_레포명.md 에 원본을 저장한다
  3. 소스 품질 평가 후 Learning Layer 시작
```

이미 Sources에 저장되어 있다면 바로 Layer 0부터 시작한다.

### Learning Layer 구조

학습은 Layer 0→3을 순서대로 진행. 페이스 변수에 따라 대기 여부 달라진다.

```
Layer 0: Bird's Eye View      ← 항상 여기서 시작
Layer 1: Problem & Value
Layer 2: Core Concepts
Layer 3: Architecture         ← Mode B는 여기서 종료가 기본
```

> **Deep Dive는 별도 세션으로**. 특정 구현 세부사항·API·엣지케이스는 Layer 3 완료 후 사용자가 명시적으로 요청하면 새 ingest 사이클(별 Source/Wiki)로 진행. Mode B 안에 억지로 Layer 4를 끼우지 않는다.

### Layer 0: Bird's Eye View

**목적**: 이 기술이 어느 영역에 속하는지, 무엇인지 한 눈에 파악한다.

```
[기술명]은 [어느 영역/생태계]에 속하는 [기술 유형]입니다.

한 줄 요약: [초등학생도 이해할 수 있는 수준의 비유]

주요 사용자: [대상]
존재하는 생태계: [관련 기술 스택이나 도메인]
```

### Layer 1: Problem & Value

**목적**: 이 기술이 해결하려는 문제와 존재 이유를 이해한다.

```
**해결하는 문제**: [기존 한계/불편함]
**왜 이 방식인가**: [트레이드오프]
**비슷한 기술 비교**:
| | [이 기술] | [대안 A] | [대안 B] |
```

> **Layer 0 + Layer 1 자동 통합 조건**: 기술 정의 자체가 문제 해결과 거의 동일하게 표현되는 경우 (예: "Claude Design — 디자이너 없이도 디자인 만들기") 두 Layer를 한 번에 제시해도 좋다.

### Layer 2: Core Concepts

**목적**: 반드시 알아야 할 3~5개의 핵심 개념을 익힌다.

- **Vault Wiki에 이미 있는 개념은 링크만 제공하고 건너뜀** (객관 기준)
- 모르는 개념은 비유와 예시로 충분히 설명한다

### Layer 3: Architecture Overview

**목적**: 전체 구조를 파악한다. 주요 컴포넌트, 동작 흐름, 텍스트 다이어그램으로 설명.

### Layer 간 확인 문구 (페이스 변수에 따라 다름)

| 페이스 | 동작 |
|------|------|
| **`step`** (default) | 각 Layer 끝에 **"Next?"** 한 단어 확인. 사용자가 "ok"/"다음"/무응답이면 진행 |
| **`rush`** | 확인 없이 Layer 0→3 한 번에 제시. 마지막에만 리뷰 요청 |
| **`partial`** | "Layer 2까지만 해보자" 같은 요청 시 해당 범위까지 진행 후 종료 |

### Wiki 작성 시점

어느 Layer에서든 사용자가 원하면 Wiki를 작성할 수 있다.
Wiki는 **현재까지 이해한 Layer까지만** 작성하고, 미완성 Layer는 TODO로 표시한다:

```markdown
## Architecture (학습 예정)
> Layer 3까지 학습 후 보강 예정
```

## MODE C: Chunked Translation Review

**언제 사용**: 공식 문서·레퍼런스·튜토리얼 등 긴 호흡의 URL인데 **원문의 정확한 워딩을 보존**하고 싶을 때.
Claude가 **묶음 크기를 자동 판단**해서 잘라 번역+요약+Claude 메모(Insight)를 병치해 제시 → 사용자가 묶음 단위로 리뷰하며 진행.

**Mode A/B와의 차이**:
- Mode A: Claude가 전체를 요약 설명 → 원문 보존 약함
- Mode B: Bird's Eye View → 단계별 심화. 원문과 거리 있음
- **Mode C: 원문을 Sources에 그대로 보존 + 자동 묶음 번역 + Insight**. 원문 정확성 + 학습 자산 동시 확보

### 사전 준비: Sources 먼저 저장

```
1. 원본 마크다운을 Sources에 저장 (Mode C 페이지당 1개)
   - 위치: Sources/docs/[도구명]/YYYY-MM-DD_페이지제목.md
   - 원문을 섹션 구분 그대로 기록 (번역·가공 전)
2. 묶음 크기 자동 판단
```

### 묶음 크기 자동 판단 규칙 ⭐

원문 길이와 구조에 따라 **Claude가 자동으로 결정**한다. 고정 "3문단" 아님.

| 섹션 성격 | 묶음 크기 |
|---------|---------|
| 섹션 전체가 3문단 이하 | **섹션 단위로 1 묶음** |
| 섹션이 긴 경우 (4문단+) | **원문 3문단씩** 쪼갬 |
| 표 (table) | **표 단위 1 묶음** (독립) |
| 코드 블록 / 설정 예제 | **블록 단위 1 묶음** |
| 리스트 (bullet list) | 7개 이하면 1 묶음, 넘으면 분할 |
| 다이어그램·이미지 | 1 묶음 |

사용자에게 분할 계획만 간단히 통보하고 바로 첫 묶음으로 진입 (TOC 선택 안 물음).
사용자가 "작게 쪼개서" / "크게 묶어서" 라고 하면 이 규칙을 override.

### 묶음 제시 루프

**Step 1 — 묶음 병치 제시** (Claude 메모 default ON)

```
─── § [섹션명] (묶음 1/N) ─────────────────

**원문**
[원문 그대로]

**번역·요약**
[한국어 번역 + 핵심 요약]

★ Insight ─────────────────────────────────────
[이 묶음에서 본인 작업(vault 도메인)과 연결되는 포인트 1-2줄.
 또는 숨은 가정·용어 함정·오해 포인트.]
─────────────────────────────────────────────────
──────────────────────────────────────────────
```

> **Claude 메모는 default ON**. 사용자가 "메모 빼" / "번역만" 라고 할 때만 OFF.

**Step 2 — 사용자 응답 처리**

| 응답 | Claude의 행동 |
|------|--------------|
| "다음" / "ok" / 무응답 | 다음 묶음으로 진행 |
| "잠깐, X 더 설명해줘" | 묶음 내 X만 확장 설명 후 계속 |
| "건너뛰어 / 스킵" | 현재 묶음 `<!-- 스킵 -->` 표시 후 다음 |
| "원문이랑 뜻이 다른 거 같은데" | 원문 재검토 + 번역 수정 |
| "이 섹션 이미 알아" | 남은 묶음 모두 번역만 제공, 리뷰 생략 |
| "묶음 크기 조정" | 자동 판단 규칙 override |

**Step 3 — Sources에 묶음 번역 추가**

원문 Sources 파일 내 섹션 아래에 추가:

```markdown
> **번역 요약 (묶음 N)**: [번역·요약 본문]
> ★ Insight: [Claude 메모 — default 포함]
```

### 전체 완료 후

```
[문서 제목] 묶음 리뷰 완료. 총 N 묶음 / 스킵 M개.

다음 단계: Wiki 작성. 어떤 경로로 할까요?
```

→ **Wiki 위치 결정** 단계로 진행.

### Sources / Wiki 저장 위치

- Sources: `Sources/docs/[도구명]/YYYY-MM-DD_페이지제목.md`
- Wiki: `Wiki/[도메인]/[개념명].md` — 도메인 라우팅 규칙에 따름

### 튜닝 변수 (사용자가 요청하면 조정)

| 변수 | 기본값 | 대안 |
|------|--------|------|
| 묶음 크기 | **자동 판단** (위 규칙) | "크게 5문단씩" / "작게 1-2문단씩" |
| 번역 상세도 | 요약형 | "직역으로" / "핵심만 불릿" |
| Claude Insight 메모 | **default ON** | "메모 빼고 번역만" |
| 페이스 | 공통 페이스 변수(`step`/`rush`/`partial`) 적용 | 상단 공통 섹션 참조 |

이 변수들은 사용자 피드백을 통해 이 vault에 최적화된 디폴트로 정착시켜 나간다.

## Wiki 위치 결정 (공통 — 작성 전 필수)

리뷰 & 수정 반영 완료 후 Wiki를 작성하기 전 위치를 확정한다.

### 자동 결정 기준 (사용자에게 묻지 않고 진행)

다음 **3개 조건이 모두 충족**되면 Claude가 자동 결정하고 **통보만** 한다:

1. **SAP 도메인 라우팅 규칙**(아래) 또는 CLAUDE.md의 도메인 표에서 해당 주제의 폴더가 이미 지정돼 있음
2. 해당 폴더 안에 동일 개념의 기존 Wiki가 **없음**(새 페이지 생성) 또는 **명백한 동일 개념 페이지가 있어 그 안에 섹션 추가**가 자명함
3. 주제가 한 도메인에만 속하고 **다른 도메인과 경계 겹침이 없음**

통보 예시:
```
Wiki 위치: Wiki/북미/Zuora/Refund.md (신규). 별 이견 없으면 작성 진행합니다.
```

### 후보 제시가 필요한 경우 (사용자 결정)

- 도메인 경계가 애매 (예: SAP vs HMG-Settlement에 걸침)
- 합치기 vs 새 파일 판단이 갈림 (기존 파일에 섹션으로 붙일지, 독립시킬지)
- 동일 주제로 추정되는 기존 파일이 2개 이상 존재

```
Wiki 페이지 위치를 결정하겠습니다.

**후보 A** — 기존 [[Wiki/도메인/기존파일명]]에 합치기
이유: (합치는 근거)

**후보 B** — 새 파일 [[Wiki/도메인/새파일명]] 생성
이유: (분리하는 근거)

**후보 C** — 기존 파일에 섹션 추가 + 교차참조
이유: (분량이 적거나 맥락이 강할 때)

어느 위치가 좋으신가요?
```

### 진행 원칙

1. 상단 **SAP 도메인 라우팅 규칙**을 먼저 확인
2. `Wiki/index.md`를 읽어 관련 도메인의 기존 페이지 목록 파악
3. 자동 결정 조건 충족 여부 판단 → 자동/후보 제시 결정
4. 자동 진행이라도 **한 줄 통보**는 반드시 (사용자가 이의 제기할 기회 유지)

## 완료 기준

- 사용자가 Claude의 1차 정리를 리뷰했고 수정 사항이 반영됐다 (모호한 부분이 있었다면 해소)
- Sources + Wiki + index.md + logs/YYYY-MM.md 가 모두 업데이트됐다
- Mode B: 학습한 레이어 범위가 wiki에 명시되어 있다 (`rush`·`partial` 페이스였다면 그 범위 기록)
- Mode C: 모든 묶음에 번역 요약 블록이 채워져 있다. Insight default ON이면 `★ Insight` 라인 포함 (스킵 포함)
