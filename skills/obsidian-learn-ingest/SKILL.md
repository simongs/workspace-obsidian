# obsidian-learn-ingest

소스를 읽고 바로 wiki를 쓰는 대신, 사용자와 대화로 이해를 맞춘 다음 wiki를 작성한다.

**이 스킬이 존재하는 이유**: LLM이 wiki를 만드는 동안 사람은 구경꾼이 된다. 기존 wiki 지식을 소스에 섞어버리는 실수도 생긴다. 대화를 통한 교차검증이 wiki 품질과 사용자의 실제 학습을 동시에 높인다.

**핵심 철학**: 소스가 부실하면 그대로 wiki로 옮기지 않는다. 먼저 사용자의 이해를 높이고 원본을 보강한 뒤, wiki로 가도 충분한 수준이 됐을 때 ingest를 진행한다.

## 소스 유형 판별 (스킬 진입 시 가장 먼저)

소스가 무엇인지 파악하고 적절한 모드를 선택한다.

| 소스 유형 | 판별 기준 | 사용 모드 |
|-----------|-----------|-----------|
| 이미지, 다이어그램, 일반 문서, 노트 | 파일 경로 / 텍스트 | **Mode A: Document Ingest** |
| GitHub 레포 / 기술 소개 URL | URL + 기술 학습 목적 | **Mode B: Tech Progressive** |
| 공식 문서 / 레퍼런스 URL | URL + "단락별로", "같이 읽어가며" 등 | **Mode C: Reference Doc Interactive** |
| **Figma URL + 문서/소스** | `figma.com/design/` URL이 포함된 경우 | **Mode D: Figma + Document Ingest** |

> URL이 주어졌을 때 "그냥 정리만" → Mode A, "새로 배우고 싶어" → Mode B, "같이 읽어가며" → Mode C.
> **Figma URL이 함께 오면 항상 Mode D** — 다른 소스 유형과 조합되더라도 우선 적용한다.
> 소스가 이미 Sources에 있다면 → Mode A로 바로 진입한다.

### 모드 판별이 애매할 때

소스 유형이나 사용자 의도가 명확하지 않으면 **직접 선택지를 제시하고 물어본다.** 임의로 모드를 선택하지 않는다.

```
어떤 방식으로 진행할까요?

**Mode A — 문서 정리** (가장 빠름)
Claude가 소스를 읽고 이해한 내용을 설명 → 사용자가 확인 → Sources + Wiki 작성.
이미 아는 내용을 기록하거나, 단순 정리가 목적일 때.

**Mode B — 기술 점진적 학습** (Bird's Eye View → 단계별 심화)
처음 접하는 기술이나 개념을 층위별로 이해하고 싶을 때.
Claude가 Layer 0(큰 그림)부터 시작해 원하는 깊이까지 안내.

**Mode C — 원문 단락별 읽기** (가장 능동적)
긴 공식 문서나 레퍼런스를 직접 읽으면서 이해하고 싶을 때.
원문을 단락씩 보여주고, 이해 내용을 실시간으로 Sources에 기록.
```

### 링크 추적 규칙 (기본 동작)

소스 파일 안에 `[[다른 파일]]` 형태의 wikilink가 있으면 **반드시 해당 파일을 모두 따라가서 함께 ingest한다.**
사용자가 별도로 요청하지 않아도 기본 동작이다.

```
사용자가 A.md를 ingest 요청
  → A.md 안에 [[B.md]], [[C.md]] wikilink 발견
  → B.md, C.md 도 함께 읽어서 통합 분석
  → 하나의 Sources + Wiki로 병합 ingest
  → A, B, C 모두 삭제
```

예외: 링크가 이미 Sources나 Wiki로 연결된 경우 (이미 ingest된 문서) → 스킵

---

### 원본 소스 처리 규칙 (ingest 완료 후)

| 소스 위치 | 처리 방법 |
|-----------|---------|
| `Daily/` | **ingest 완료 후 파일 삭제** (Daily = 스테이징 영역. Source에 원본 전문 callout 포함) |
| `Notes/Inbox/` | **ingest 완료 후 파일 삭제** (처리 대기 스테이징 영역) |
| `Notes/` (Inbox 제외) | 파일 보존. 하단에 역참조 추가: `related: [[Sources/...]], [[Wiki/개념명]]` |

## 소스 품질 평가 (모든 모드 공통)

모드 진입 전, 소스를 읽은 뒤 wiki로 바로 옮겨도 충분한 수준인지 평가한다.

**부실한 소스의 징표**:
- 단편적인 메모, 키워드 나열, 미완성 문장
- "이게 뭔지 궁금하다", "나중에 정리" 같은 의도만 있고 내용이 없음
- 핵심 개념의 설명 없이 용어만 등장
- 작성자 본인도 완전히 이해하지 못한 상태로 남긴 흔적

**부실하다고 판단되면 → 보강 모드 진입**:

```
소스를 읽었습니다. 내용이 아직 단편적이어서 wiki로 바로 옮기기보다
먼저 이 주제를 함께 탐구한 뒤 더 풍부한 상태로 기록하는 게 좋겠습니다.

[주제에 대한 심화 설명 — Claude가 알고 있는 것을 풍부하게 가르쳐 줌]

이해가 됐으면 추가로 궁금한 점을 물어보세요.
충분히 이해됐다고 느껴지면 그때 Sources + Wiki를 작성하겠습니다.
```

보강 과정:
1. Claude가 먼저 주제를 폭넓게 설명 (소스에 없는 내용도 포함, 단 출처 명시)
2. 사용자가 이해를 확인하고 추가 질문
3. **"이 정도면 wiki로 가도 좋겠다"는 합의가 이뤄질 때** Phase 2 또는 해당 모드로 진행

**충분하다고 판단되면 → 해당 모드 본 단계로 바로 진행**.

## MODE A: Document Ingest Mode (기본)

**언제 사용**: 이미지, 다이어그램, Daily 노트, Notes, 일반 문서 등을 ingest할 때. 가장 자주 쓰이는 기본 모드.

### Phase 1: 소스 분석 (사용자에게 알리기 전에 조용히)

소스 파일을 Read 도구로 읽는다. 이미지라면 반드시 Read로 직접 열어서 시각적으로 확인한다.

읽으면서 스스로 정리:
- 이 소스가 다루는 **핵심 주제** 는 무엇인가?
- **섹션이나 구분**이 있다면 각각 무엇을 설명하는가?
- **모호하거나 확인이 필요한 부분**은 어디인가?
- 기존 vault wiki와 연결될 개념은 무엇인가?

> 주의: 이 단계에서는 기존 wiki 지식을 소스 내용에 섞지 않는다. 소스에 있는 내용만 정리한다.

→ 소스 품질 평가 후 충분하면 Phase 2로, 부실하면 보강 모드 진입.

### Phase 2: 사용자에게 이해한 내용 설명

분석한 내용을 사용자에게 섹션별로 설명한다. 형식:

```
[소스 제목 / 파일명] 을 읽었습니다. 제가 이해한 내용을 섹션별로 설명드리겠습니다.

**① [섹션명]**
[이 섹션에서 이해한 내용. 이미지라면 다이어그램/화면에서 읽은 것만.]

**② [섹션명]**
[...]

---
확인해주실 사항:
- 제가 잘못 이해한 부분이 있나요?
- 추가로 설명해주실 내용이 있나요?
```

**절대 하지 않는 것**: 기존 wiki 페이지에서 알고 있는 내용을 소스 설명에 끼워 넣기.
소스에 없는 내용이 필요하다면 "(이 부분은 소스에 없어서 기존 wiki 기준으로 추가한 내용입니다)" 라고 명시한다.

### Phase 3: Q&A 교차검증

사용자 응답을 받고 Q&A를 진행한다.

**Claude → 사용자 (모호한 부분 확인)**
- 소스에서 명확히 읽히지 않은 부분
- 기술 용어나 약어의 의미
- 다이어그램의 화살표 방향이나 레이블 뜻
- 예외 케이스나 전제 조건

**사용자 → Claude (이해 확인)**
- Claude가 틀리게 이해한 부분 수정
- 소스에 없지만 연관된 맥락 추가
- 더 깊이 다루고 싶은 개념

Q&A는 자연스럽게 진행하되, 충분히 이해가 맞춰졌다고 느껴지면:

```
이제 충분히 내용을 파악한 것 같습니다.
Wiki를 작성할까요? 작성 전에 다른 확인하실 사항이 있으면 말씀해주세요.
```

### 팁: 이미지 소스일 때

이미지는 텍스트 문서보다 오독 가능성이 높다. Phase 2에서 반드시:
- 섹션 제목(■, ①, 번호 등)을 그대로 인용해서 설명
- 다이어그램의 흐름은 화살표 방향대로 순서를 설명
- 읽기 어려운 텍스트는 "이 부분은 명확히 읽히지 않았습니다" 라고 솔직하게 표시

## MODE B: Tech Progressive Learning Mode

**언제 사용**: 새로운 기술, 라이브러리, 도구, 프레임워크를 처음 접하는 경우.
기반 지식이 없어도 괜찮다. Bird's Eye View부터 시작해 단계적으로 이해한다.

> 이미 어느 정도 아는 기술이라도 "새로 공부하고 싶어"라면 Mode B 사용 가능.
> "그냥 README를 정리만 해줘"라면 Mode A.

### 사전 준비: 소스 확보

```
URL이 주어진 경우:
  1. WebFetch 또는 defuddle로 페이지를 읽는다
  2. Sources/github/YYYY-MM-DD_레포명.md 에 원본을 저장한다
  3. 소스 품질 평가 후 Learning Layer 시작
```

이미 Sources에 저장되어 있다면 바로 Layer 0부터 시작한다.

### Learning Layer 구조

학습은 레이어를 순서대로 진행한다. **각 레이어가 충분히 이해된 다음에만 다음으로 넘어간다.**
사용자가 원하는 깊이까지만 진행하고 wiki를 작성해도 된다.

```
Layer 0: Bird's Eye View      ← 항상 여기서 시작
Layer 1: Problem & Value      ← Layer 0 이해 확인 후
Layer 2: Core Concepts        ← Layer 1 이해 확인 후
Layer 3: Architecture         ← Layer 2 이해 확인 후
Layer 4: Deep Dive            ← 선택적, 사용자 요청 시
```

### Layer 0: Bird's Eye View

**목적**: 이 기술이 어느 영역에 속하는지, 무엇인지 한 눈에 파악한다.

```
[기술명]은 [어느 영역/생태계]에 속하는 [기술 유형]입니다.

한 줄 요약: [초등학생도 이해할 수 있는 수준의 비유]

주요 사용자: [대상]
존재하는 생태계: [관련 기술 스택이나 도메인]
```

확인: "이해되셨나요? Layer 1 (왜 만들어졌는지)로 넘어갈까요?"

### Layer 1: Problem & Value

**목적**: 이 기술이 해결하려는 문제와 존재 이유를 이해한다.

```
**해결하는 문제**: [기존 한계/불편함]
**왜 이 방식인가**: [트레이드오프]
**비슷한 기술 비교**:
| | [이 기술] | [대안 A] | [대안 B] |
```

확인: "납득되셨나요? Layer 2 (핵심 개념)로 넘어갈까요?"

### Layer 2: Core Concepts

**목적**: 반드시 알아야 할 3~5개의 핵심 개념을 익힌다.

- 사용자가 이미 아는 개념은 건너뛴다
- 모르는 개념은 비유와 예시로 충분히 설명한다

확인: "더 설명 필요한 개념이 있나요? Layer 3 (전체 구조)로 넘어가겠습니다."

### Layer 3: Architecture Overview

**목적**: 전체 구조를 파악한다. 주요 컴포넌트, 동작 흐름, 텍스트 다이어그램으로 설명.

확인: "전체 흐름이 이해되셨나요? Layer 4로 더 깊이 들어가거나, 여기서 Wiki를 작성할까요?"

### Layer 4: Deep Dive (선택적)

특정 구현 세부사항, API, 설정, 엣지케이스. 사용자 주도 대화로 진행.

### Wiki 작성 시점

어느 레이어에서든 사용자가 원하면 wiki를 작성할 수 있다.
wiki는 **현재까지 이해한 레이어까지만** 작성하고, 미완성 레이어는 TODO로 표시한다:

```markdown
## Architecture (학습 예정)
> Layer 3까지 학습 후 보강 예정
```

## MODE C: Reference Doc Interactive Learning

**언제 사용**: 공식 문서, 레퍼런스 페이지, 튜토리얼 등 긴 호흡의 URL을 학습할 때.
단락을 하나씩 함께 읽으면서 이해하고, Sources가 학습 중 자연스럽게 만들어진다.

**Mode B와 다른 점**:
- Mode B: Claude가 문서를 읽고 레이어별로 설명 → 사용자가 듣는 구조
- Mode C: 사용자가 단락을 직접 읽고 해석 → Claude가 보조. 능동적 학습.

### 사전 준비: Sources 먼저 저장 후 시작

URL이든, 사용자가 직접 붙여넣은 마크다운이든 동일하게:

```
1. 원본 전체 마크다운을 먼저 Sources에 저장한다
   - 위치: Sources/docs/[도구명]/YYYY-MM-DD_페이지제목.md
   - 각 섹션 끝에 학습 메모 자리 표시: <!-- 💡 이해: 학습 예정 -->
2. TOC를 사용자에게 보여주고 건너뛸 섹션을 선택받는다
3. 문서의 맨 처음(인트로)부터 시작한다 — 섹션 1이 아님
```

TOC 형식:
```
(인트로) 도입부
① [섹션 1]
② [섹션 2]
...
건너뛰고 싶은 항목이 있으면 알려주세요. 없으면 인트로부터 시작합니다.
```

### 단락별 인터랙션 루프

**Step 1 — 원문 제시**
```
─── [인트로 / 섹션명] ───────────────────
[원문 텍스트 그대로]
────────────────────────────────────────────
읽어보셨나요? 이해한 내용을 말씀해주세요.
```

**Step 2 — 사용자 응답 처리**

| 응답 | Claude의 행동 |
|------|--------------|
| 이해한 내용을 말함 | 확인 + 보완 + Sources `💡 이해` 블록 업데이트 |

| "이미 알아" | 간단히 확인 후 넘어감 |

## MODE D: Figma + Document + Code Ingest Mode

**언제 사용**: `figma.com/design/` URL이 제공된 경우.
Figma 기획 화면 + 문서(선택) + 코드베이스(선택)를 통합해 wiki를 만든다.

> Figma URL만 단독으로 오는 경우도 Mode D로 처리한다.

### Step 0: Figma URL 파싱

URL에서 `fileKey`와 `nodeId`를 추출한다.

```
figma.com/design/{fileKey}/{파일명}?node-id={nodeId}
→ nodeId의 "-"를 ":"으로 변환 (예: 8791-4656 → 8791:4656)
```

### Step 1: 소스 수집 (병렬 실행)

**A. Figma 데이터 수집**
```
→ get_design_context(fileKey, nodeId) 호출
→ 응답이 메타데이터만 오면 주요 서브레이어를 추가 호출
```

**B. 문서 소스 읽기** (있는 경우)
```
→ 파일 경로라면 Read 도구로 읽기
→ URL이라면 WebFetch / defuddle로 읽기
```

**C. 코드 탐색** ← **entity 이름이 주어지면 항상 수행**

entity 이름이 제공되면 **Explore 서브에이전트**를 사용해 코드를 탐색한다.
Explore는 격리된 컨텍스트에서 탐색 후 요약만 반환하므로 메인 컨텍스트를 오염시키지 않는다.

```
Explore 서브에이전트 프롬프트 예시:
  "{코드저장소경로}에서 {EntityName} 관련 코드를 탐색해줘.
   - domain 레이어: entity 클래스 (컬럼, 타입, 제약조건), PolicyService (주요 메서드)
   - application 레이어: SearchService, UseCaseService (주요 메서드)
   - API 레이어(선택): 관련 Controller 엔드포인트
   thoroughness: medium"
```

hmg-billing 코드베이스의 경우 표준 경로 패턴:
```
domain/.../domain/{entity}/entity/{Entity}Entity.kt
domain/.../domain/{entity}/{Entity}PolicyService.kt
application/.../application/{entity}/{Entity}SearchService.kt
application/.../application/{entity}/{Entity}UseCaseService.kt
```

코드 탐색으로 얻을 수 있는 정보:
- 실제 컬럼 타입 및 길이
- UNIQUE 제약조건, 인덱스
- 수정 가능/불가 필드 (entity의 update() 메서드 확인)
- Figma에 없는 필드 발견 (예: `aip_integration_mode`)
- API 엔드포인트 목록
- 캐시 전략
- 암호화 여부

### Step 2: 세 소스 교차 분석

읽은 내용을 속으로 정리한다:

**Figma에서 파악할 것:**
- 화면/컴포넌트 이름 및 계층 구조
- 주요 UI 요소 (검색 조건, 버튼, 테이블 컬럼, 상태값 등)
- 화면 설명 패널에 기재된 기획 노트 (`Description` 프레임)
- 수정 가능/불가 필드 (UI 기획 기준)

**문서에서 파악할 것:**
- 비즈니스 규칙 및 정책
- Figma에는 없는 예외 케이스, 전제 조건
- 용어 정의 및 코드값

**코드에서 파악할 것:**
- 실제 컬럼 타입, 길이, 제약조건
- Figma/문서에 없는 필드 (코드에만 존재)
- 서비스 레이어 메서드 목록 (어떤 작업이 가능한가)
- 기술 구현 세부사항 (캐싱, 암호화, 감사 로그)

**교차 검증:**
- Figma UI ↔ 코드 컬럼 매핑
- 문서 규칙 ↔ 코드 구현 일치 여부
- Figma에만 있는 요소, 코드에만 있는 필드 → Q&A에서 확인

## Callout 스타일 가이드 (obsidian-admonition)

Wiki 문서 작성 시 ``` 코드블록 대신 **Obsidian Native Callout**을 사용한다.
실제 프로그래밍 코드(Java, Kotlin, SQL, bash 명령어)는 ``` 코드블록을 유지한다.

### 타입 매핑

| 상황 | Callout 타입 | 사용 예 |
|------|:----------:|------|
| 흐름도, 아키텍처 다이어그램, 분개 패턴 | `[!example]` | 시스템 구성도, GL 분개 |
| 핵심 원칙 / 반드시 기억해야 할 결론 | `[!important]` | "CCS는 Zuora만 바라보면 된다" |
| 배경 설명 / 개념 보충 | `[!info]` | 개념 흐름, 모델 설명 |
| 판별 로직 / 멘탈 모델 / 빠른 참조 | `[!tip]` | 구독 유형 판별, 2쌍 구조 |
| 주의사항 / 혼동 방지 / SAP 핵심 제약 | `[!warning]` | Credit Memo 주의, Chargeback, 이중 처리 방지 |
| 정상 케이스 / 성공 시나리오 | `[!success]` | Scenario A — 정상 결제 |
| 미확인 / 열린 질문 | `[!question]` | +1 Document 미파악 |
| 후속 작업 / 미결 사항 | `[!todo]` | Phase 2 대체 작업 |
| 이슈 / 문제 원인 | `[!failure]` | 버그 원인, 에러 케이스 |
| 학습 인사이트 / 핵심 발견 | `[!insight]` | 세션 아카이브의 ★ Insight |
| 원본 인용 | `[!quote]` | 원본 노트 전문 callout |

### 원칙

1. **모든 callout은 기본 펼침** — `-` (접힘) 접미사 사용 금지
   - **예외**: 원본 전문 인용(`[!quote]-`)은 내용이 길어 기본 접힘을 허용한다
2. **실제 코드는 코드블록 유지** — Java, Kotlin, SQL, bash 등 언어 지정된 코드블록은 변환하지 않는다
3. **텍스트 아트 흐름도는 callout으로** — `→ ↓ ←` 화살표, `[시스템명]` 패턴, ASCII 박스 등
4. **단순 1줄 흐름은 callout 안 인라인 코드로** — `` `A → B → C` `` 형태
5. **과도한 callout 금지** — 모든 문단을 감싸지 않는다. 의미있는 강조에만 사용

### 변환 예시

*Before*:
````
```
[고객] → [Zuora] → [PaymentTech] → [카드사]
```
````

*After*:
```markdown
> [!example] HMG 북미 결제 흐름
> `[고객]` → `[Zuora]` → `[PaymentTech]` → `[카드사]`
```

## Wiki 작성 (공통)

사용자 위치 선택 후 CLAUDE.md Ingest 워크플로우를 따른다.

**Sources 저장 위치 기준**:

| 소스 유형 | Sources 경로 |
|-----------|---------------|
| GitHub 레포 / 기술 URL (Mode B) | `Sources/github/YYYY-MM-DD_레포명.md` |
| 공식 문서 / 레퍼런스 (Mode C) | `Sources/docs/[도구명]/YYYY-MM-DD_페이지제목.md` (학습 중 이미 완성) |
| 일반 외부 소스 | `Sources/외부-YYYY-MM-DD-제목.md` |
| Daily에서 ingest | `Sources/daily/YYYY-MM-DD_제목.md` |
| Notes에서 ingest | `Sources/notes-YYYY-MM-DD-제목.md` |

```
0. (선행) Sources 파일이 이미 존재하면 읽어서 기존 `wiki:` 확인
1. Sources 저장 (Mode C는 이미 완료 — 건너뜀)
   - Daily / Notes/Inbox 소스: **제목 바로 아래, 정제 내용보다 먼저** 원본 전문을 callout으로 포함
     > [!quote]- 원본 노트 전문 보기
     > (원본 파일 전체 내용. 포맷 보존, 내용 변경 없음.)
   원본 callout 아래에 정제된 내용(Q&A 검증 완료 문구, 섹션들) 배치
2. Wiki/[도메인]/[개념명].md 생성 또는 업데이트
   - 기존 wiki 페이지와 wikilink 연결
3. 영향 받는 기존 Wiki 페이지에 교차참조 추가
4. Wiki/index.md 업데이트
5. 원본 소스 처리 (상단 "원본 소스 처리 규칙" 참조)
6. Wiki/logs/YYYY-MM.md에 항목 추가: [날짜] #ingest | 제목 (출처: github/docs/daily/notes/외부)
7. Wiki/log.md 인덱스 갱신
8. Source 파일의 `wiki:` 프로퍼티 업데이트 (양방향 링크 완성)
   - 생성/업데이트된 Wiki 페이지 경로를 Source frontmatter에 추가
   - `wiki:` 프로퍼티가 없으면 새로 추가, 있으면 항목 추가
   - Wiki 파일의 `sources:` 프로퍼티도 이 Source을 가리키는지 확인
```

## 완료 기준

- 사용자가 Claude의 소스 이해를 검증했다
- Q&A로 모호한 부분이 해소됐다
- Sources + Wiki + index.md + logs/YYYY-MM.md 가 모두 업데이트됐다
- Mode B: 학습한 레이어 범위가 wiki에 명시되어 있다
- Mode C: 모든 섹션에 `💡 이해` 블록이 채워져 있다 (건너뜀 포함)
- Mode D: Sources에 `figma-url`, `figma-node` 프로퍼티가 기록됐다. Figma 화면 구성 + 문서 규칙 + 코드 분석이 Wiki에 통합되어 있다. entity 이름이 주어진 경우 Explore 서브에이전트 코드 탐색 결과가 Sources에 포함됐다
